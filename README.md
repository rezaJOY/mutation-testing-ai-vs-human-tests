# Evaluating the Quality of AI-Generated Unit Tests Using Mutation Coverage

**Author:** Reza  
**Tool:** Mutmut 2.4.4 · **LLM:** Groq `llama-3.3-70b-versatile` · **Language:** Python

---

## Research Question

> *Do AI-generated unit tests detect injected faults as effectively as human-written tests, as measured by mutation coverage?*

---

## Overview

This study benchmarks the fault-detection quality of AI-generated pytest tests against human-written tests across three popular open-source Python libraries. The primary metric is **mutation score** — the percentage of injected code mutations (bugs) killed by the test suite — produced by [Mutmut 2.4.4](https://mutmut.readthedocs.io/).

AI tests are generated by passing the raw source file of each target module to a Groq-hosted LLM. Human tests are the unmodified test suites shipped with each project. Both are evaluated against the same set of mutants on the same source module.

---

## Project Selection

Three projects were selected based on the following criteria, applied uniformly:

| Criterion | Requirement |
|---|---|
| Language | Pure Python |
| Target module LOC | 200–600 (non-comment, non-blank lines) |
| Existing test suite | Pytest-compatible, passes cleanly |
| Dependency footprint | Self-contained — no heavy internal cross-module imports |
| Maintenance status | Actively maintained on GitHub |

> **Why these three:** `psf/black` was evaluated and rejected. `linegen.py` imports from nine internal black submodules (`black.brackets`, `black.comments`, `black.nodes`, etc.). AI-generated tests cannot import these without installing black as a full package, making the generation step non-reproducible from source file alone. `joke2k/faker` was substituted — it was already on the original candidates shortlist.

| Project | Module Tested | LOC* | GitHub |
|---|---|---|---|
| `dbader/schedule` | `schedule.py` | verify with `vet_project.py` | https://github.com/dbader/schedule |
| `mahmoud/boltons` | `boltons/strutils.py` | verify with `vet_project.py` | https://github.com/mahmoud/boltons |
| `joke2k/faker` | `faker/providers/internet/__init__.py` | verify with `vet_project.py` | https://github.com/joke2k/faker |

*\*LOC figures must be confirmed by running `vet_project.py` before committing to these modules. If any module falls outside 200–600 LOC, select an alternative from the same repo.*

> **Scope note:** Three projects were used due to the computational cost of mutation testing on a MacBook A1708 (8 GB RAM). Mutmut runs are CPU-bound and take 30–90 minutes per module. This is explicitly acknowledged as a limitation under Threats to Validity.

## Methodology

### LLM Input

The LLM receives **the source file content only** — no surrounding repo context, no imports from other modules, no existing tests. This mirrors the realistic use case of a developer pasting a module into an AI tool and matches the reproducibility requirement: anyone can replicate the generation step by providing the same file.

This is an intentional asymmetry. Human tests were written with full knowledge of the codebase; the AI receives only the file. This asymmetry is not a flaw — it is a finding and is reported as a known structural confound.

### Generation Prompt

The following prompt is used verbatim for every module, with `{source_code}` replaced by the file contents:

```
You are a Python test engineer writing pytest unit tests.

Module to test:
{source_code}

Requirements:
- Use pytest style (def test_...)
- Each test must have a clear assert statement
- Cover: normal inputs, edge cases, invalid inputs
- Do not use mocks unless absolutely necessary
- Import only from the standard library and pytest
- Return only valid Python code, no explanations.
```

The prompt is fixed and identical across all three projects. Prompt variation is not treated as an independent variable in this study.

### Retry and Discard Policy

Each generation attempt is cleaned (markdown fences and prose stripped). If no `def test_` is found in the cleaned output, one retry is made. If the retry also fails, the module is marked as **discarded** and excluded from results.

### Validation

Before any mutmut run, generated tests are checked with `validate_tests.py`, which confirms:

1. `pytest --collect-only` returns exit code 0
2. At least one test is collected
3. No `SyntaxError` or `ImportError` is present
4. At least one `assert` statement exists in the file

Tests that fail validation are discarded.

### Mutmut Configuration

A `setup.cfg` is added to each cloned project root before any mutmut run. The exact content per project:

**schedule:**
```ini
[mutmut]
paths_to_mutate = schedule.py
runner = python -m pytest tests/
tests_dir = tests/
dict_synonyms =
```

**boltons:**
```ini
[mutmut]
paths_to_mutate = boltons/strutils.py
runner = python -m pytest tests/
tests_dir = tests/
dict_synonyms =
```

**faker:**
```ini
[mutmut]
paths_to_mutate = faker/providers/internet/__init__.py
runner = python -m pytest tests/
tests_dir = tests/
dict_synonyms =
```

For the AI test run, the runner is overridden via CLI to point at `ai_tests/`:

```bash
mutmut run --paths-to-mutate <module_path> --runner "python -m pytest ai_tests/"
```

> **macOS note:** `--parallel` is a boolean flag with no argument. Use `mutmut run --parallel`, not `mutmut run --parallel 4`.

### Cache Management

Mutmut writes scores to `.mutmut-cache` in the project root. Running mutmut a second time in the same directory resumes or overwrites the same cache. **Human and AI runs must use separate cache files.** Extract and archive scores after each run before proceeding.

## Reproducing Results

### 1. Clone this repository

```bash
git clone https://github.com/rezaJOY/mutation-testing-ai-vs-human-tests.git
cd mutation-testing-ai-vs-human-tests
```

### 2. Create and activate a virtual environment

```bash
python3 -m venv .venv
source .venv/bin/activate
```

> Always activate this environment before running any script or mutmut command. Every subsequent command assumes the venv is active.

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

`requirements.txt` is generated via `pip freeze` after environment setup. `mutmut==2.4.4` is hard-pinned.

### 4. Set your Groq API key

```bash
export GROQ_API_KEY="your_key_from_console.groq.com"
echo $GROQ_API_KEY  # verify it is set — must not be empty
```

The key is never hardcoded in any script. `.gitignore` excludes `.env` and `*.env`.

### 5. Clone target projects

```bash
git clone https://github.com/dbader/schedule projects/schedule
git clone https://github.com/mahmoud/boltons projects/boltons
git clone https://github.com/joke2k/faker projects/faker
```

### 6. Record metadata (commit hashes + tool versions)

```bash
python3 scripts/record_metadata.py
```

This creates `results/metadata.json` with exact commit hashes for all three cloned projects, Python version, mutmut version, and LLM model string.

### 7. Vet projects and confirm modules

```bash
python3 scripts/vet_project.py projects/schedule
python3 scripts/vet_project.py projects/boltons
python3 scripts/vet_project.py projects/faker
```

Confirm that the target module for each project falls within 200–600 LOC and that pytest collects cleanly. If a module is outside range, select an alternative before proceeding.

### 8. Write setup.cfg per project

Copy the exact `setup.cfg` content from the Mutmut Configuration section above into each project root. Verify the file exists before running mutmut.

### 9. Generate AI tests

```bash
python3 scripts/generate_tests.py \
  projects/schedule/schedule.py \
  projects/schedule/ai_tests/test_ai_schedule.py

python3 scripts/generate_tests.py \
  projects/boltons/boltons/strutils.py \
  projects/boltons/ai_tests/test_ai_strutils.py

python3 scripts/generate_tests.py \
  projects/faker/faker/providers/internet/__init__.py \
  projects/faker/ai_tests/test_ai_internet.py
```

Raw LLM responses are saved to `results/raw_ai_outputs/` before cleaning.

### 10. Validate AI tests

```bash
python3 scripts/validate_tests.py projects/schedule/ai_tests/test_ai_schedule.py
python3 scripts/validate_tests.py projects/boltons/ai_tests/test_ai_strutils.py
python3 scripts/validate_tests.py projects/faker/ai_tests/test_ai_internet.py
```

Any file that returns `INVALID` is discarded. Do not proceed to mutmut for that module.

### 11. Run mutmut — human tests

Run from inside each project directory. Extract and archive scores immediately after each run.

```bash
# schedule
cd projects/schedule
mutmut run --parallel
cd ../..
python3 scripts/extract_scores.py projects/schedule/.mutmut-cache human schedule
cp projects/schedule/.mutmut-cache results/schedule_human.mutmut-cache
rm projects/schedule/.mutmut-cache

# boltons
cd projects/boltons
mutmut run --parallel
cd ../..
python3 scripts/extract_scores.py projects/boltons/.mutmut-cache human boltons
cp projects/boltons/.mutmut-cache results/boltons_human.mutmut-cache
rm projects/boltons/.mutmut-cache

# faker
cd projects/faker
mutmut run --parallel
cd ../..
python3 scripts/extract_scores.py projects/faker/.mutmut-cache human faker
cp projects/faker/.mutmut-cache results/faker_human.mutmut-cache
rm projects/faker/.mutmut-cache
```

### 12. Run mutmut — AI tests

Cache has been cleared. Now run against `ai_tests/`:

```bash
# schedule
cd projects/schedule
mutmut run --paths-to-mutate schedule.py --runner "python3 -m pytest ai_tests/" --parallel
cd ../..
python3 scripts/extract_scores.py projects/schedule/.mutmut-cache ai schedule

# boltons
cd projects/boltons
mutmut run --paths-to-mutate boltons/strutils.py --runner "python3 -m pytest ai_tests/" --parallel
cd ../..
python3 scripts/extract_scores.py projects/boltons/.mutmut-cache ai boltons

# faker
cd projects/faker
mutmut run --paths-to-mutate faker/providers/internet/__init__.py --runner "python3 -m pytest ai_tests/" --parallel
cd ../..
python3 scripts/extract_scores.py projects/faker/.mutmut-cache ai faker
```

## Threats to Validity

**Selection Bias.** Modules were selected based on a documented LOC range (200–600), pytest-compatibility, and self-contained import structure. This may favour modules with simpler, well-isolated functions. Mitigation: criteria are documented and applied uniformly across all three projects.

**Prompt Sensitivity.** AI-generated tests are sensitive to prompt wording. A different prompt structure may produce significantly different mutation scores. This study uses a single fixed prompt template across all modules. Prompt variation is left as future work.

**Test Count Asymmetry.** AI generation typically produces more tests per module than human-written baselines. Higher test volume may independently improve mutation scores regardless of individual test quality. Test counts are reported in `results.csv` and must be considered when interpreting score differences.

**Structural Advantage of Human Tests.** Human tests were written with direct knowledge of the implementation. The AI receives only the source file. This is an unavoidable and intentional asymmetry — it reflects the realistic deployment context of AI test generation tools and is explicitly stated as a known limitation, not an experimental error.

**Small Sample Size.** Three projects is below the threshold for statistical generalization. Significance testing is not applied. Findings should be treated as a directional pilot study, not a definitive benchmark.
