# Evaluating the Quality of AI-Generated Unit Tests Using Mutation Coverage

**Drafted by:** Reza  
**Tool:** Mutmut 2.4.4 · **LLM:** Groq `llama-3.3-70b-versatile` · **Language:** Python

---

## Research Question

> *Do AI-generated unit tests detect injected faults as effectively as human-written tests, as measured by mutation coverage?*

---

## Overview

This study benchmarks the fault-detection quality of AI-generated pytest tests against human-written tests across three popular open-source Python libraries. The primary metric is **mutation score** — the percentage of injected code mutations (bugs) killed by the test suite — produced by [Mutmut 2.4.4](https://mutmut.readthedocs.io/).

AI tests are generated by passing the raw source file of each target module to a Groq-hosted LLM. Human tests are the unmodified test suites shipped with each project. Both are evaluated against the same set of mutants on the same source module.

## Methodology

### LLM Input

The LLM receives **the source file content only** — no surrounding repo context, no imports from other modules, no existing tests. This mirrors the realistic use case of a developer pasting a module into an AI tool and matches the reproducibility requirement: anyone can replicate the generation step by providing the same file.

This is an intentional asymmetry. Human tests were written with full knowledge of the codebase; the AI receives only the file. This asymmetry is not a flaw — it is a finding and is reported as a known structural confound.

### Generation Prompt

The following prompt is used verbatim for every module, with `{source_code}` replaced by the file contents:

```
You are a Python test engineer writing pytest unit tests.

Module to test:
{source_code}

Requirements:
- Use pytest style (def test_...)
- Each test must have a clear assert statement
- Cover: normal inputs, edge cases, invalid inputs
- Do not use mocks unless absolutely necessary
- Import only from the standard library and pytest
- Return only valid Python code, no explanations.
```

The prompt is fixed and identical across all three projects. Prompt variation is not treated as an independent variable in this study.

### Retry and Discard Policy

Each generation attempt is cleaned (markdown fences and prose stripped). If no `def test_` is found in the cleaned output, one retry is made. If the retry also fails, the module is marked as **discarded** and excluded from results.

### Validation

Before any mutmut run, generated tests are checked with `validate_tests.py`, which confirms:

1. `pytest --collect-only` returns exit code 0
2. At least one test is collected
3. No `SyntaxError`, `ImportError`, or `ModuleNotFoundError` is present
4. At least one `assert` statement exists in the file

Tests that fail validation are discarded.

### Mutmut Configuration

A `setup.cfg` is added to each cloned project root before any mutmut run. Exact content per project:

**schedule:**
```ini
[mutmut]
paths_to_mutate = schedule/
runner = python3 -m pytest test_schedule.py
tests_dir = .
dict_synonyms =
```

**boltons:**
```ini
[mutmut]
paths_to_mutate = boltons/strutils.py
runner = python3 -m pytest tests/
tests_dir = tests/
dict_synonyms =
```

**jmespath:**
```ini
[mutmut]
paths_to_mutate = jmespath/lexer.py
runner = python3 -m pytest tests/
tests_dir = tests/
dict_synonyms =
```

For the AI test run, the runner is overridden via CLI:

```bash
mutmut run --paths-to-mutate <module_path> --runner "python3 -m pytest ai_tests/"
```

> **macOS note:** `--parallel` is a boolean flag — no argument. Use `mutmut run --parallel`, never `mutmut run --parallel 4`.

### Cache Management

Mutmut writes scores to `.mutmut-cache` in the project root. Running mutmut a second time in the same directory resumes or overwrites the same cache. **Human and AI runs must use separate cache files.** Extract and archive scores after each run before proceeding.

## Reproducing Results

### 1. Clone this repository

```bash
git clone https://github.com/rezaJOY/mutation-testing-ai-vs-human-tests.git
cd mutation-testing-ai-vs-human-tests
```

### 2. Create and activate a virtual environment

```bash
python3 -m venv .venv
source .venv/bin/activate
```

> Always activate this environment before running any script or mutmut command.

### 3. Install pipeline dependencies

```bash
pip install -r requirements.txt
```

### 4. Set your Groq API key

```bash
export GROQ_API_KEY="your_key_from_console.groq.com"
echo $GROQ_API_KEY  # must not be empty
```

### 5. Clone target projects

```bash
git clone https://github.com/dbader/schedule projects/schedule
git clone https://github.com/mahmoud/boltons projects/boltons
[git clone https://github.com/jmespath/jmespath.py projects/jmespath.py](https://github.com/jmespath/jmespath.py projects/jmespath.py)
```

### 6. Install target projects in editable mode

> **Critical:** Do not skip this step. Installing in editable mode makes Python import directly from the cloned source files. Without this, mutmut mutates the source but the test suite imports from pip (site-packages) — mutations are never executed and all scores will be wrong.

```bash
pip install -e projects/schedule
pip install -e projects/boltons
pip install -e projects/jmespath.py
```

### 7. Record metadata

```bash
python3 scripts/record_metadata.py
```

Saves `results/metadata.json` with commit hashes, tool versions, and model string. Run this before any mutmut run.

### 8. Vet projects

```bash
python3 scripts/vet_project.py projects/schedule
python3 scripts/vet_project.py projects/boltons
python3 scripts/vet_project.py projects/jmespath.py
```

Confirm target module for each project is within 200–600 LOC and pytest collects cleanly.

### 9. Write setup.cfg per project

Copy the exact `setup.cfg` content from the Mutmut Configuration section above into each project root.

### 10. Generate AI tests

```bash
python3 scripts/generate_tests.py \
  projects/schedule/schedule/__init__.py \
  projects/schedule/ai_tests/test_ai_schedule.py

python3 scripts/generate_tests.py \
  projects/boltons/boltons/strutils.py \
  projects/boltons/ai_tests/test_ai_strutils.py

python3 scripts/generate_tests.py \
  projects/jmespath.py/jmespath/lexer.py \
  projects/jmespath.py/ai_tests/test_ai_lexer.py
```

### 11. Validate AI tests

```bash
python3 scripts/validate_tests.py projects/schedule/ai_tests/test_ai_schedule.py
python3 scripts/validate_tests.py projects/boltons/ai_tests/test_ai_strutils.py
python3 scripts/validate_tests.py projects/jmespath.py/ai_tests/test_ai_lexer.py
```

Any `INVALID` result — do not proceed to mutmut for that module.

### 12. Run mutmut — human tests

Run from repo root. Do not `cd` into the project directory.

```bash
# schedule
cd projects/schedule && mutmut run --parallel && cd ../..
python3 scripts/extract_scores.py projects/schedule/.mutmut-cache human schedule
cp projects/schedule/.mutmut-cache results/schedule_human.mutmut-cache
rm projects/schedule/.mutmut-cache

# boltons
cd projects/boltons && mutmut run --parallel && cd ../..
python3 scripts/extract_scores.py projects/boltons/.mutmut-cache human boltons
cp projects/boltons/.mutmut-cache results/boltons_human.mutmut-cache
rm projects/boltons/.mutmut-cache

# jmespath
cd projects/jmespath.py && mutmut run --parallel && cd ../..
python3 scripts/extract_scores.py projects/jmespath.py/.mutmut-cache human jmespath
cp projects/jmespath.py/.mutmut-cache results/jmespath_human.mutmut-cache
rm projects/jmespath.py/.mutmut-cache
```

### 13. Run mutmut — AI tests

```bash
# schedule
cd projects/schedule && mutmut run --paths-to-mutate schedule/ --runner "python3 -m pytest ai_tests/" --parallel && cd ../..
python3 scripts/extract_scores.py projects/schedule/.mutmut-cache ai schedule
cp projects/schedule/.mutmut-cache results/schedule_ai.mutmut-cache
rm projects/schedule/.mutmut-cache

# boltons
cd projects/boltons && mutmut run --paths-to-mutate boltons/strutils.py --runner "python3 -m pytest ai_tests/" --parallel && cd ../..
python3 scripts/extract_scores.py projects/boltons/.mutmut-cache ai boltons
cp projects/boltons/.mutmut-cache results/boltons_ai.mutmut-cache
rm projects/boltons/.mutmut-cache

# jmespath
cd projects/jmespath.py && mutmut run --paths-to-mutate jmespath/lexer.py --runner "python3 -m pytest ai_tests/" --parallel && cd ../..
python3 scripts/extract_scores.py projects/jmespath.py/.mutmut-cache ai jmespath
cp projects/jmespath.py/.mutmut-cache results/jmespath_ai.mutmut-cache
rm projects/jmespath.py/.mutmut-cache
```

## Threats to Validity

**Selection Bias.** Modules were selected based on a documented LOC range (200–600), pytest-compatibility, and self-contained import structure. This may favour modules with simpler, well-isolated functions. Mitigation: criteria are documented and applied uniformly.

**Prompt Sensitivity.** AI-generated tests are sensitive to prompt wording. A different prompt structure may produce significantly different mutation scores. This study uses a single fixed prompt template. Prompt variation is left as future work.

**Test Count Asymmetry.** AI generation typically produces more tests per module than human-written baselines. Higher test volume may independently improve mutation scores regardless of individual test quality. Test counts are reported in `results.csv`.

**Structural Advantage of Human Tests.** Human tests were written with direct knowledge of the implementation. The AI receives only the source file. This is an unavoidable and intentional asymmetry — stated as a known limitation, not an experimental error.

**Small Sample Size.** Three projects is below the threshold for statistical generalization. Significance testing is not applied. Findings should be treated as a directional pilot study, not a definitive benchmark.
